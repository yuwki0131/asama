# 並列開始前の基盤整備(直列で実施)

並列化の成功条件は「**分割リファクタリング先行 → 並列開始**」の順序を守ること。
現状のコード構造のまま5エージェントを走らせると、以下の巨大単一ファイルで確実に衝突する。

## 1. 巨大単一ファイルの分割

| ファイル | 現状 | 分割案 | 実施者 |
|---|---|---|---|
| `packages/simulation/src/index.ts` | 2,345行 | combat / pathfinding / economy / food / ai / serialization 等 | シミュレーション |
| `apps/game/src/client/renderer/GameCanvas.tsx` | 1,559行 | 描画レイヤー単位 | UI/基盤 |
| `assets/source/blender/scripts/render_asset.py` | 約3,000行 | 共通基盤(カメラリグ・契約・マテリアル・レンダー設定)+ ドメイン別モジュール(建築・地形系 / ユニット・植生系) | アートA |
| `assets/definitions/production-assets.json` | 2,304行 | カテゴリ別に分割(建築・地形系 / ユニット・植生系) | アートA |

分割は挙動を変えないリファクタリングとして行い、既存テスト+typecheck+
アセット監査・キャリブレーションの全パスを完了条件とする。

**render_asset.py 分割の必須完了条件(アート並列化の成立条件)**:
レンダーキャッシュのキーを「スクリプト全文のハッシュ」から
「共通基盤モジュール+当該モデルが属するドメインモジュールのハッシュ」へ変更すること
(`packages/asset-tools/src/renderCache.ts` の `computeRenderCacheKey`)。
これを欠くと、アートBのユニットビルダー1行の変更でアートAの全建築・地形キャッシュが
無効化され、マージごとに全再レンダー(25分)+全PNGのchurnが発生し、
A/Bの分離が実質機能しない。なお、ペイントオーバー済みアセット(建物・植生の大半)は
raster取り込みでBlenderレンダーを通らないため、スクリプト結合の影響面は
「接続キット+地形+ユニット」に限定される — 分割の実務コストは半日規模。

各分割はそれぞれの所有エージェント(の最初のタスク)として実施してよいが、
**同領域の機能追加より先に完了させる**こと。半日〜1日規模の投資で以降の
マージコストが桁で変わる。

- [ ] `.gitignore` から `future/` を外し、本ディレクトリをコミット対象化
      (現状は ignore 対象のため、ワークスペース間で同期されない)

## 3. 実施順序の目安

1. `.gitignore` 解除 + future/ コミット(方針の共有が最優先)
2. CI ワークフロー(マージゲートがないと以降の統合が危険)
3. 各所有エージェントによる巨大ファイル分割(並列可: sim / ui / art-a は互いに独立)
4. ブランチ・クローン・requests 雛形の整備
5. 並列開発の開始
