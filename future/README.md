## 背景: なぜ並列化するか

MVP完成・アセットパイプライン成熟という節目に対し、残作業は次の4軸に整理できる。

- **A. ビジュアル完成**: ペインタリー移行(2026-07-04承認)の全面適用、ユニットアニメ、天守Blender化
- **B. ゲームプレイ深化**: MVPから繰り延べた項目の回収(兵種相性・高低差・射線・攻撃側兵糧・鉄砲/騎兵)
- **C. コンテンツ多様化**: AIプロファイル追加、シナリオ複数化
- **D. 技術基盤**: 巨大単一ファイルの分割、UI層テスト

これらを直列で進めると時間がかかるため、担当領域を排他的に分けた
**5エージェント並列体制**で進める。D(技術基盤)は独立エージェントとせず、
各エージェントが自領域のリファクタリングを最初のタスクとして持つ。

## 実行者モデル(2026-07-05確定)

- **フォアグラウンド**: アートA/B+統括は**メインセッション(統括エージェント)**が持つ。
  アートの判定はユーザーとの対話式(小出しリテイク前提)であり、この直列性は不可避のため
  セッションの主戦場とする。1往復に複数の判定案件を載せて往復回数を圧縮する。
- **バックグラウンド**: sim / UI / コンテンツは**機械ゲート(typecheck・test・監査)で
  自走可能**なため、統括が codex exec を別クローン・別ブランチでバックグラウンド起動し、
  ユーザーの判定待ち時間と並行して走らせる。統括は毎ターン裏レーンを回収・PR統合する。
- 将来、持続性が必要になったら(日をまたぐ自走等)ユーザー運転の別セッション or
  定期実行へ昇格する(クローン+PR設計は実行者非依存のため移行コストなし)。
