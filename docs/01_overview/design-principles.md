# 設計原則

## 1. Stronghold風の単純さを優先する

複雑なシミュレーションより、プレイヤーが状況を理解して即座に操作できることを優先します。

- 建築は即時
- 住民は自律行動
- 兵士は個兵単位で操作
- 固定部隊や厳密な陣形は持たない
- 味方同士の重複を許容
- 敵ユニットは通過不可

## 2. 日本城郭らしさは勝敗構造で表現する

見た目だけでなく、以下をゲームルールとして成立させます。

- 門が強襲の主要突破点
- 石垣・土塁は通常歩兵では実質破壊不能
- 梯子と堀埋めが代替経路
- 本丸が攻撃側の最終目標
- 城内蔵と兵糧が籠城能力を決める

## 3. 史実よりゲーム上の整合性を優先する

- 正確な年代区分は設けない
- 天守の様式を自由に選択可能
- 建物、兵種、城郭形式をマッシュアップ可能
- 歴史的雰囲気を保ちつつ例外処理は削減する

## 4. 食料以外の物流は抽象化する

- 食料だけ蔵・荷車・接続経路を持つ
- 木材、石材、武器、銭は勢力共有値
- 生産施設から貯蔵庫への実物流は実装しない
- 補給演出はゲーム判定と分離する

## 5. MVPでは追加しやすい要素を後回しにする

次の要素はMVP外です。

- 火災
- 大筒・輸入大砲
- Fog of War
- 天候
- 士気
- 疲労
- 寺社
- 技術ツリー
- 実在武将
- マルチプレイヤー

## 6. 調整値はデータ駆動にする

以下はコードに固定せず、定義データまたはシナリオデータとして変更可能にします。

- HP、攻撃力、射程、攻撃間隔
- 兵種相性
- 建物コスト・HP
- 堀埋め時間
- 蔵容量
- 食料消費
- 収穫量
- 敵襲時刻・編成
- 補給荷車の積載量・出現条件

## 7. 未確定事項を推測で固定しない

Coding Agentは、未確定事項について次の順で対応します。

1. 既存仕様から矛盾なく導ける最小実装を選ぶ
2. パラメータ化する
3. `unresolved-issues.md`へ記録する
4. 将来拡張を阻害しない構造にする
